### [BUG-016] Custom HTML sanitization uses regex ‚Äî bypassable XSS vectors
- **File**: `backend/index.js` lines ~6539‚Äì6551, `app/book/[id]/page.tsx` lines ~903, ~1003, ~1008
- **What**: Custom HTML sanitization uses regexes to strip `<script>`, `<iframe>`, event handlers. Regex-based HTML sanitization is fundamentally bypassable. Examples: `<img src=x onerror = alert(1)>` (space before `=`), `<svg/onload=alert(1)>`, `<a href="&#106;avascript:alert(1)">`, HTML encoding tricks. The frontend renders custom_html via `dangerouslySetInnerHTML` and custom_css via `<style dangerouslySetInnerHTML>`.
- **Impact**: Workspace owners on custom plan could inject XSS that executes on every visitor's browser on the public booking page. Mitigated by: only custom-plan owners can set it, and it only runs on their own booking page.
- **Note**: This is a defense-in-depth concern. For a proper fix, use a DOM-based sanitizer like DOMPurify on the frontend before rendering.
- **Fix owner**: AI 2 (frontend rendering) + AI 1 (backend sanitization)
- **Status**: NEW

### [BUG-017] `decHtml` in dashboard uses `innerHTML` on textarea ‚Äî potential XSS

- **What**: The `decHtml` function creates a `<textarea>` element, sets its `innerHTML` to the input string, then reads `.value`. If the input contains `<img src=x onerror=alert(1)>`, the browser may parse and execute it when setting `innerHTML`, even on a detached textarea.

- **Note**: Modern browsers don't execute scripts in detached textareas, but the pattern is still flagged as a concern.
- **Fix owner**: AI 2
