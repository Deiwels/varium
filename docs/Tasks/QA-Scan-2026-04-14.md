## POTENTIAL ISSUE

### [SMS-001] /api/verify/send and /api/verify/check require auth but should be public
- **File**: `backend/index.js` — new endpoints `POST /api/verify/send`, `POST /api/verify/check`
- **What**: These endpoints are under `/api/` path which has `authenticate + requireAuth + resolveWorkspace` middleware. But OTP verification is typically used BEFORE login (for signup, password reset, phone verification during booking).
- **Impact**: Unauthenticated users can't send/check OTP codes — the endpoints will return 401.
- **Fix**: Move to `/public/verify/send` and `/public/verify/check`, OR create a middleware exception for these routes.
- **Fix owner**: AI 1 / Owner
- **Status**: NEW — needs verification (depends on intended use case)
- **Note**: If OTP is only for authenticated users (e.g., changing phone number in settings), then `/api/` is correct. If it's for login/signup, needs to be `/public/`.

- **Impact**: Not a bug, but code duplication. If one is updated and the other forgotten, they'll diverge.
