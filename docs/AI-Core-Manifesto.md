## 5 Золотих Правил

### Правило 1: ЧИТАЙ ПЕРЕД ТИМ ЯК РОБИТИ
> *Read First, Act Second*

- Перед будь-якою роботою читай `docs/Home.md`, `docs/Tasks/In Progress.md` та `docs/Tasks/4-AI-Remaining-Work-Split.md`
- Перевіряй `git log --oneline -10` і `git diff HEAD --name-only` — щоб знати що вже зроблено
- Шукай у `docs/` чи вже є рішення для твоєї задачі
- **Ніколи не дій наосліп.** Якщо не впевнений — спочатку читай, потім запитуй, потім кодуй

---

### Правило 2: НЕ ЛАМАЙ ТЕ, ЩО ПРАЦЮЄ
> *Fix, Don't Rewrite*

- Якщо функціонал вже готовий і задеплоєний — **не переписуй його по-новому**
- Ми виправляємо конкретні зафіксовані проблеми, а не покращуємо все підряд
- Перед зміною будь-якого існуючого коду — прочитай його і зрозумій навіщо він написаний саме так
- Не додавай нові фічі поки не попросив Owner — навіть якщо здається що "так буде краще"

---

### Правило 3: ПОВАЖАЙ СВІЙ СКОУП
> *Strict Ownership — No Trespassing*

- **AI 1** не чіпає `app/`, `components/`, `lib/`
- **AI 2** не чіпає `backend/index.js`, `.github/workflows/`
- **AI 3 (Verdent)** не пише продуктовий код без погодженого плану — тільки планує і верифікує
- **AI 4** втручається в будь-який файл тільки при emergency — і одразу повідомляє інших

Якщо потрібна крос-скоупова зміна:
1. AI 3 складає план
2. Owner затверджує
3. Відповідний AI (1 або 2) реалізує у своєму файлі

---

### Правило 4: ДОКУМЕНТУЙ КОЖНУ ДРІБНИЦЮ
> *Obsidian Log Rule — If it's not written, it doesn't exist*

- Будь-яка виконана робота — навіть 2-рядковий фікс — має бути записана в `docs/DevLog/YYYY-MM-DD.md`
- Нові задачі → `docs/Tasks/In Progress.md`
- Завершені задачі → відмічай `[x]` і переноси в `In Progress.md` розділ "Done"
- Нові баги → `docs/Tasks/QA-Scan-YYYY-MM-DD.md` (AI 3 веде, але будь-який AI може додати)
- Це єдиний спосіб не робити одну роботу двічі

---

### Правило 5: БЕЗ ПЛАНУ НЕМАЄ КОДУ
> *Plan-Driven Execution — No Surprises*

- Будь-яка **масштабна** зміна (нова фіча, рефактор, нова інтеграція, зміна архітектури) вимагає плану від AI 3
- Що значить "масштабна": зачіпає 3+ файли, або змінює поведінку існуючих ендпоінтів, або впливає на дані клієнтів
- Дрібні фікси (1-2 файли, очевидна помилка) AI 1 або AI 2 можуть робити самостійно — але документують у DevLog
- **Ніхто не приймає архітектурних рішень одноосібно.** Навіть якщо дуже впевнений

## Процес роботи (Flow)

```
Owner повідомляє про задачу
        ↓
AI 3 читає docs → перевіряє git → складає детальний план
        ↓
Owner (або всі AI) погоджують план
        ↓
AI 1 і AI 2 реалізують паралельно у своїх зонах
        ↓
AI 3 верифікує результат → оновлює QA Scan і Docs
        ↓
Owner робить live перевірку
        ↓
Повторюємо
```

## Що робити при Emergency (коли щось впало в продакшні)

1. **AI 4** отримує задачу від Owner — фіксить швидко
2. AI 4 одразу записує що змінив у `docs/DevLog/YYYY-MM-DD.md`
3. AI 3 після фіксу перевіряє коректність і оновлює QA Scan
4. AI 1 або AI 2 роблять clean версію фіксу у своєму скоупі якщо потрібно

## Заборонені дії (Hard Rules — ніколи)

| Дія | Чому заборонено |
|---|---|
| Редагувати чужий скоуп без погодження | Викликає конфлікти, зламані функції, дублювання |
| Переписувати готовий і працюючий код | Створює нові баги, витрачає час |
| Комітити без запису в DevLog | Інші AI не знають що змінилось |
| Починати масштабну роботу без плану | Гарантує конфлікти і регресії |
| Ігнорувати цей документ | Руйнує всю систему співпраці |

## Контакт між AI

Оскільки AI не спілкуються напряму — **єдиний канал координації це `docs/`**.

- Хочеш повідомити іншому AI про проблему → запиши в `QA-Scan` або `In Progress.md`
- Хочеш передати задачу → запиши в `In Progress.md` з чітким власником
- Хочеш підтвердити що твоя частина готова → запиши в `DevLog` і відмич `[x]` в `In Progress.md`

## Session Start Protocol (Обов'язковий чеклист старту)

**Кожен AI виконує ці 5 кроків перед початком будь-якої роботи — без винятків:**

```
Крок 1: git log --oneline -10
         → що змінилось з останньої сесії, хто комітив

Крок 2: git diff HEAD --name-only
         → чи є незакомічені зміни від інших AI в поточному дереві

Крок 3: Читати docs/Tasks/In Progress.md
         → що зараз офіційно в роботі, що вже закрито

Крок 4: Читати docs/DevLog/YYYY-MM-DD.md (сьогоднішній)
         → що зробили інші AI сьогодні

Крок 5: Перевірити останній QA-Scan файл (docs/Tasks/QA-Scan-*.md)
         → чи є нові баги або блокери
```

Тільки після цих 5 кроків — починати роботу. Якщо пропустити — ризик дублювання або зламати щось вже готове.

## Commit Standards (Стандарт комітів)

Всі 4 AI використовують один формат — щоб `git log` був читабельним для всіх:

```
Формат: <type>(<scope>): <short description>

type:   feat | fix | docs | refactor | chore | hotfix
scope:  backend | frontend | sms | auth | billing | docs | element

Приклади:
  feat(backend): add phone_number_index for O(1) STOP lookup
  fix(frontend): show business details without phone input
  docs(sms): update Element 10DLC resubmission checklist
  hotfix(backend): stripe webhook after auth middleware — 401
```

**Правила комітів:**
- Кожен коміт = одна атомарна зміна (не мішати backend + frontend в одному)
- Запис в `DevLog` обов'язковий **до** коміту — не після
- Не комітити незавершений код (якщо робота в процесі — WIP коміт тільки в окрему гілку)
- `hotfix` коміти завжди позначати в DevLog міткою `[HOTFIX]`

## Security Rules (Правила безпеки)

Ці правила незмінні незалежно від терміновості задачі:

| Правило | Деталь |
|---|---|
| Не логувати секрети | Ніколи `console.log(apiKey)`, `console.log(token)`, `console.log(password)` |
| Не хардкодити credentials | API ключі, phone numbers клієнтів, паролі — тільки через env vars |
| Не видаляти дані з Firestore | Без явного підтвердження Owner'а — навіть тестові документи |
| Не відкривати endpoint без auth | Нові `/api/` ендпоінти завжди за auth middleware. Виняток тільки `/public/` з дозволу |
| Не обходити sanitizeHtml | Вже є в коді — не замінювати на raw string конкатенацію |
| Сумнів = зупинка | При будь-якому сумніві щодо безпеки — зупинись, запитай AI 3 + Owner |

## Rollback Protocol (Що робити коли щось зламалось)

```
Рівень 1 — Баг в dev / щось не так локально:
  → AI 1 або AI 2 фіксить у своєму скоупі самостійно
  → Записує в DevLog

Рівень 2 — Production регресія (фіча перестала працювати):
  → AI 3 оцінює масштаб і повідомляє Owner'а
  → Якщо швидкий фікс (1-2 файли): AI 1 або AI 2 фіксить
  → Якщо складно або крос-скоупово: AI 4 робить emergency hotfix
  → Записується в DevLog з міткою [HOTFIX]

Рівень 3 — Critical failure (дані, payments, auth, data loss):
  → СТОП. Жоден AI нічого не чіпає.
  → Owner приймає рішення — тільки він
  → При потребі: git revert <commit> (тільки з дозволу Owner'а)
  → AI 4 робить hotfix виключно після "go" від Owner'а
```

## Escalation Triggers (Коли зупинитись і запитати Owner'а)

AI **зобов'язаний зупинитись** і повідомити Owner перед тим як діяти, якщо задача зачіпає:

- **Платіжні дані** — Stripe webhook, Apple IAP, payroll розрахунки
- **Auth / JWT / паролі** — будь-яка зміна в механізмі авторизації
- **Видалення даних** — будь-який `delete` або `set` що перезаписує в Firestore
- **Element Barbershop** — protected legacy workspace, не чіпати без явного OK
- **Публікація закритого endpoint'а** — перетворення `/api/` на `/public/`
- **Конфлікт між AI** — якщо два AI дійшли до протилежних рішень
- **Вихід за межі плану** — задача стала більшою ніж описано в погодженому плані

Якщо сумніваєшся — це вже trigger. Краще зупинитись і запитати, ніж зламати.

## Decision Log (Журнал архітектурних рішень)

Всі важливі рішення фіксуються в `docs/Architecture/Decision-Log.md`.

**Формат:**
```
## [DECISION-XXX] Назва рішення
Date: YYYY-MM-DD
Status: Accepted | Deprecated
Context: Чому виникло питання
Decision: Що вирішили
Alternatives considered: Що відкинули і чому
Consequences: Що це означає для майбутнього
```

Перед тим як прийняти нове архітектурне рішення — перевір Decision Log. Можливо воно вже було прийнято або відкинуто з вагомих причин.
