### Hotfix: Waitlist disappeared from public booking page — SET-006 regression from 849e998 (commit `a3c885f`)

**Symptom:** Owner reported that the `Join waitlist` CTA on the public booking page (`/book/{slug}`) stopped rendering. It was there earlier in the week and then silently vanished after 2026-04-15 backend pushes.

**Root cause** — my own earlier SET-006 fix in commit `849e998`:
- In `/public/config/:workspace_id` I started returning `waitlist_enabled: !!data.waitlist_enabled`. The `!!` coerced *undefined* → *false*.
- In 99% of live Salon+ workspaces the `waitlist_enabled` field was **never explicitly set** in Firestore — the `Settings → Waitlist enabled` toggle defaulted to unchecked in the UI but no one had touched it. So every Salon+ workspace suddenly got `waitlist_enabled: false` from the public config endpoint.
- Frontend at `app/book/[id]/page.tsx:442` uses `setWaitlistEnabled(cfg?.waitlist_enabled ?? !!resolved.waitlist_enabled)`. The nullish coalescing `??` only falls through on `null` / `undefined`, not on `false`. So the explicit `false` from config overrode the plan-feature `true` from `/public/resolve`, and the form disappeared everywhere.

**Two other silent bugs uncovered** during the investigation, both already dead before my 849e998 fix — my change just made one of them visible:
- `POST /public/waitlist/:workspace_id` at line ~9400 rejected submissions with `403 Waitlist is currently disabled` whenever `settingsData.waitlist_enabled` was falsy, including undefined. So even before 849e998, every "click Submit" on an untouched-toggle workspace failed with 403. Form was visible but non-functional. Predates my work.
- `tryWaitlistAutoFill()` at line ~810 bailed immediately on every cancellation if `!settings.waitlist_enabled`. The auto-notify feature (cancel slot → tell waitlist) was effectively dead for every Salon+ workspace that never touched the toggle. Also predates my work.

**Fix applied in `a3c885f`** — one consistent semantic across all three sites: *waitlist is ON by default on any plan that includes the feature; the Settings toggle is an explicit OFF override, not an opt-in.*

1. **`/public/config/:workspace_id`** — only emits `waitlist_enabled` when it is an explicit boolean in Firestore. When absent, the field is omitted from the response, and the frontend `??` guard correctly falls through to the plan-feature value from `/public/resolve`.
2. **`POST /public/waitlist/:workspace_id`** — rejects only when the plan does not include waitlist OR when the owner explicitly set `waitlist_enabled === false`. Undefined (default) passes.
3. **`tryWaitlistAutoFill()`** — bails only on explicit `false`. Plan-level gate is already enforced in the caller path.

**Behavior matrix after the fix:**

| Scenario | Before `a3c885f` | After `a3c885f` |
|---|---|---|
| Salon+ workspace, toggle untouched (default) | Form hidden (regression from 849e998), submit would 403 anyway | ✅ Form visible, submit accepted |
| Salon+ workspace, explicit `waitlist_enabled: true` | Form visible, submit accepted | ✅ Same — unchanged |
| Salon+ workspace, explicit `waitlist_enabled: false` | Form hidden, submit rejected | ✅ Same — explicit off respected |
| Individual plan | Plan gate blocks regardless | ✅ Same — plan gate unchanged |
| Element Barbershop | Not affected by this path | ✅ Not affected |

**Re-verification required** after `a3c885f` deploy lands on Cloud Run — owner and Codex need to run together:

- [ ] Open `/book/{salon-workspace-slug}` in incognito → pick a fully booked day → `Join waitlist` CTA visible
- [ ] Fill waitlist form → submit → **no 403 error** → success state
- [ ] Admin `/waitlist` page shows the new entry
- [ ] Cancel an existing booking → check `workspaces/{wsId}/waitlist` for matching entries → SMS + email auto-notify fires (or email-only fallback if SMS sender is not active)
- [ ] Salon+ workspace with explicit `waitlist_enabled: false` in Firestore → form still hidden, POST still returns 403
- [ ] Individual plan workspace → form hidden (plan gate)
- [ ] Element Barbershop public page (`/book/elementbarbershop`) → Business details + Services preview still render (e97efd9 functionality unaffected)
- [ ] Run full `docs/Tasks/Element-10DLC-Resubmission-Checklist.md` top to bottom on Element

**Lesson learned.** Boolean defaults in public API responses are load-bearing. When converting an "omitted" field to an "explicit boolean" field, check every frontend consumer for `??` / `||` fallback patterns. The fix should either keep the field optional (omit when unset) OR explicitly think through what "unset" should mean and document the default. Both pre-existing backend call sites (`POST /public/waitlist/` and `tryWaitlistAutoFill`) were already making the wrong assumption — the feature "worked" before only because the frontend form was visible and users happened not to try submit on unset workspaces. My 849e998 change exposed the full chain. Treat this as a reminder to audit **all** consumers when flipping a default, not just the direct caller.

### Telnyx Hardening — Backend Gaps 2, 3, 4, 5 (backend/index.js)

Implemented the full safe-hardening pack from `docs/Tasks/Telnyx-Integration-Plan.md`. Sender architecture unchanged — still per-workspace toll-free + grandfathered manual 10DLC + Element protected. The product change is purely in owner experience (auto-provisioning removes the "Enable SMS" friction) plus backend security/perf.

**Gap 2 — Ed25519 webhook signature verification**
- New helper `verifyTelnyxWebhookSignature(req)` — mirrors the Stripe/Square pattern already in the codebase
- No-op when `TELNYX_WEBHOOK_PUBLIC_KEY` env var is absent (dev/staging-safe; does not block before the secret is rotated in)
- Called on entry of both `POST /api/webhooks/telnyx` (inbound SMS) and `POST /api/webhooks/telnyx-10dlc` (brand/campaign status updates)
- Uses `crypto.verify(null, ...)` with Ed25519 SPKI key from base64 env var
- On failure: throws `{ statusCode: 401 }` which short-circuits webhook processing before any opt-out / reminder writes happen
- Operational gate: `TELNYX_WEBHOOK_PUBLIC_KEY` is now owner-confirmed in GitHub Secrets; the remaining step is confirming a deployed backend revision is actually enforcing webhook verification in production

**Gap 3 — `phone_number_index` for O(1) inbound webhook lookup**
- New top-level Firestore collection `phone_number_index/{e164Digits}` with fields `workspace_id`, `shop_name`, `from_number`, `number_type`, `updated_at`
- Written from both provisioning paths: `provisionTollFreeSmsForWorkspace()` after successful toll-free provision, and `POST /api/sms/verify-otp` after successful manual 10DLC SP campaign activation
- `POST /api/webhooks/telnyx` now resolves the inbound TO number via a single `phone_number_index` doc read (was: full workspace scan with N settings reads), and propagates STOP opt-out via `collectionGroup('clients').where('phone_norm', '==', ...)` instead of iterating every workspace. Pending `sms_reminders` cancellation is set-based on affected workspaces only
- Removes the hard `limit(100)` workspace cap and makes STOP/HELP latency independent of total workspace count

**Gap 4 — `runAutoReminders()` pagination**
- Replaced `db.collection('workspaces').limit(100).get()` with a `startAfter(lastDoc)` pagination loop ordered by `__name__`, batch size 50, no upper cap per cycle. Throttle (3 min) stays in place via `_lastReminderRun`

**Gap 5 — SMS auto-provisioning on plan activation**
- New non-throwing helper `autoProvisionSmsOnActivation(workspaceLike, source)` with 4 guards: legacy/protected (isLegacyManualSmsPath), already-active, provisioning-in-flight, and failed_max_retries. Only proceeds from statuses `none`, `rejected`, `failed`
- On success: clears retry metadata and records `sms_auto_provision_succeeded_at`
- On failure: increments `sms_auto_provision_retry_count`, stores `sms_auto_provision_last_error`, schedules `sms_auto_provision_next_retry_at` with exponential backoff (5 min → 15 min → 45 min → 2 h → 6 h). After 5 attempts: sets `sms_registration_status: 'failed_max_retries'`. Writes audit log entries `sms.auto_provision_failed` / `sms.auto_provision_exhausted` per attempt. Never throws — fire-and-forget callers
- Wired into three activation points: `POST /auth/signup` (source `signup_trial`), `handleStripeEvent()` for `checkout.session.completed` / `customer.subscription.created|updated` / `invoice.payment_succeeded`, and `POST /api/billing/apple-verify` (source `apple_paid`)
- New background job `runSmsAutoProvisionRetry()` wired into the main `setInterval` (3 min tick). Throttled to 5 min. Paginated workspace iteration (same pattern as Gap 4) with safety cap of 500 workspaces per cycle. Deliberately avoided `collectionGroup('settings')` to not require an extra Firestore index exemption. Fires `autoProvisionSmsOnActivation(..., 'auto_retry')` for any workspace whose `sms_auto_provision_next_retry_at` is due and status is not already terminal

### Smart Booking Audit System (`runBookingAudit()`)
Background job running every 4 hours, scans all workspaces for booking anomalies.

**8 Health Checks:**

| # | Check | Severity | Action |
|---|-------|----------|--------|
| 1 | Double bookings (overlapping times, same barber) | critical | Alert |
| 2 | Ghost barber (deleted/inactive barber on future booking) | critical | SMS + email to client |
| 3 | Stale "booked" status (past >2h) | warning | **Auto-fix to noshow** |
| 4 | Missing client data (no name, phone, or email) | warning | Alert |
| 5 | Orphaned bookings (expired workspace plan) | critical | Alert |
| 6 | Schedule violations (booking outside working hours) | warning | Alert |
| 7 | Cancellation spike (>50% in 24h, min 5 bookings) | warning | Alert |
| 8 | Repeat no-show clients (3+ in 30 days) | info | Alert |

**Notifications:**
- `support@vurium.com` — digest of all critical issues across all workspaces
- Workspace owner — push notification + branded dark-cosmos email
- In-app badge — stored in `settings/booking_audit` doc per workspace

### Auto-Fix Stale Bookings
- Past bookings with `status: 'booked'` (>2 hours overdue) automatically changed to `noshow`
- Adds `auto_noshow: true` + `auto_noshow_at` timestamp for audit trail
- Previously only warned; now self-heals

### Ghost Barber Client Alerts
When audit detects a future booking assigned to a deleted/inactive barber:
- **SMS** to client: "Your specialist is no longer available... rebook here"
- **Email** to client: branded template with "Reschedule" + "Book Another Specialist" buttons
- Dedup via `ghost_barber_notified` flag on booking doc — won't spam on next audit cycle

### Smart Client Recommendations (`getSmartRecommendation()`)
When a public booking fails, the API now returns actionable alternatives:
- On `409 Conflict` — next 3 available slots for same barber + up to 3 alternative barbers
- On `400 Outside Schedule` — next available slots within working hours
- On `404 Barber Not Found` — other active barbers with their nearest open slot

Response includes `recommendation` object:
```json
{
  "error": "Slot already booked",
  "recommendation": {
    "message": "This time is unavailable. Here are the nearest open slots:",
    "alternative_slots": ["2026-04-15T15:00:00Z", "..."],
    "alternative_barbers": [{ "id": "abc", "name": "Mike", "next_slot": "..." }]
  }
}
```

Enhanced endpoints:
- `POST /public/bookings/:workspace_id` (conflict, outside schedule, barber not found)
- `POST /public/manage-booking/reschedule` (conflict, outside schedule)

### Client Satisfaction Ping + Google Reviews
After a booking is marked `done`/`completed`:
- **Email immediately** — branded "How was your visit?" with star rating icons + "Leave a Google Review" button
- **SMS 2 hours later** — "Thanks for your visit! Leave a review here: {google_review_url}"
- Dedup: won't schedule if satisfaction reminder already exists for that booking
- Respects SMS opt-out via existing `runAutoReminders()` pipeline

**New settings:**
- `google_review_url` — owner saves their Google Review link
- `satisfaction_sms_enabled` — toggle on/off (defaults to on)

### Waitlist Auto-Fill
When any booking is cancelled (3 paths: PATCH status, DELETE, public cancel):
1. Checks if the cancelled slot is in the future and actually free
2. Queries waitlist for matching `barber_id` + `date` + preferred time window
3. Sends first matching waitlist client SMS + branded email: "A spot just opened up!"
4. Marks waitlist entry as `notified: true, auto_fill: true`
- Reuses existing waitlist notification email template (party icon + table + "Book Now" CTA)

### Booking Rate Limiter
- In-memory sliding window: max 3 bookings per phone/email per hour per workspace
- Returns `429 Too many booking requests`
- Applied to `POST /public/bookings/:workspace_id` and `POST /public/bookings-group/:workspace_id`
- Records hit only after successful booking (not on validation errors)
- Auto-cleanup of expired entries every 10 minutes

### Booking Audit Status Endpoint
- `GET /api/booking-audit/status` — returns last audit results from `settings/booking_audit`
- Fields: `last_run`, `warnings_count`, `critical_count`, `warnings[]`
- Pattern matches existing `GET /api/payroll/audit/status`
