### iOS app auth bootstrap hotfix — native cookie mismatch caused black screen / endless sign-in

Owner reported a critical regression in the native iOS app (the `WKWebView` wrapper around `vurium.com`):

- already logged-in users saw a black screen
- fresh sign-in inside the app could get stuck in an endless loading loop

Investigation across the web repo and the native Swift wrapper found a contract mismatch, not a rendering-only bug.

**Web expectations**

- `middleware.ts` guarded app routes using only the canonical role cookie `VURIUMBOOK_TOKEN`
- `components/Shell.tsx` expected `localStorage.VURIUMBOOK_TOKEN` to exist immediately on first paint; otherwise it redirected to `/signin`

**Native iOS reality**

In `/Users/nazarii/Desktop/untitled folder/VuriumBook/VuriumBook/VuriumWebView.swift`, cold-start boot still seeds the web session with:

- `vuriumbook_auth` = `role:uid`
- `vuriumbook_token` = bearer token

That means the native app was restoring a legacy cookie contract while the website had moved on to the canonical `VURIUMBOOK_TOKEN` contract. The result was:

1. `WKWebView` opens `/dashboard`
2. `middleware.ts` does not see `VURIUMBOOK_TOKEN` and redirects to `/signin`
3. native/user-script restore races with the web bootstrap
4. `Shell.tsx` sees no `localStorage.VURIUMBOOK_TOKEN` yet and marks the session as `noauth`
5. user gets trapped in a black-screen / signin loop

**Hotfix landed in the web repo**

- `middleware.ts`
  - accepts legacy `vuriumbook_auth` as a fallback role cookie
  - mirrors it forward into canonical `VURIUMBOOK_TOKEN` so the session self-heals
- `components/Shell.tsx`
  - restores `localStorage.VURIUMBOOK_TOKEN` from legacy `vuriumbook_token` before declaring `noauth`
- `lib/auth-cookie.ts`
  - writes both `VURIUMBOOK_TOKEN` and `vuriumbook_auth`
  - clears `VURIUMBOOK_TOKEN`, `vuriumbook_auth`, and `vuriumbook_token` on logout so native stale sessions do not resurrect themselves

**Second half of the loop (caught after the first hotfix).**

The owner then reported a more specific live symptom: the iOS app briefly showed the main screen and instantly threw the user back into loading, over and over, making the sign-in form unusable. Claude's review branch identified the missing piece:

- on `401`, `lib/api.ts` cleared `localStorage` but did **not** reliably kill the JS role cookie
- `middleware.ts` still saw that cookie and redirected `/signin` back to `/dashboard`
- Safari / WKWebView were also ignoring our cookie delete in some cases because the original cookie was set with `Secure` and the delete path did not always match attributes

Follow-up fix:

- `lib/api.ts` now calls `clearAuthCookie()` on any non-login `401`
- `lib/auth-cookie.ts` now includes the `Secure` attribute on delete when running under HTTPS, matching how the cookie was originally set
- `lib/api.ts` and `components/Shell.tsx` now invoke the native `logout` bridge on forced auth teardown, which makes `VuriumWebView.swift` clear `UserDefaults` and prevents the native wrapper from re-injecting the same expired JWT into `localStorage` on the next `/signin` load

### BE.1 — Distributed lock for background jobs (`withJobLock()`)

**Problem.** The `setInterval(() => { ... }, 3 * 60 * 1000)` block at `backend/index.js:10257` dispatched 7 background jobs every 3 minutes without any cross-instance coordination. When Cloud Run scales to multiple instances under load (cold start while warm instance is still serving, load spike, Cloud Run auto-scale), every instance fires all 7 jobs simultaneously — duplicate SMS reminders, duplicate auto-notify emails, duplicate Telnyx provision attempts, duplicate payroll audits, duplicate booking audits, double Anthropic spend on AI diagnostic scans. Currently `min_instances=0` so the risk is dormant most of the time, but it becomes real any time Cloud Run spins up a second warm instance.

**Fix.** New `withJobLock(jobName, ttlSeconds, fn)` helper uses Firestore top-level `job_locks/{jobName}` collection as a distributed TTL lock. Each Cloud Run instance generates a `JOB_INSTANCE_ID` (`crypto.randomBytes(8).toString('hex')`) once at module load. Before running a job, the helper opens a Firestore transaction that:
1. Reads the lock doc
2. If it exists with `locked_until > now` AND `locked_by !== our instance id` → skips this cycle
3. Otherwise → writes the lock with our instance id and `locked_until = now + ttlSeconds`

After the job completes, a second transaction deletes the lock *only if we still own it* (guards against stomping on a replacement holder that grabbed the lock after TTL). If the release transaction fails, the lock expires naturally via TTL (within at most 15 minutes for the longest-running jobs).

**Jobs wrapped** (at `backend/index.js:10342`):

| Job | TTL |
|---|---|
| `runAutoReminders` | 600 s |
| `runAutoMemberships` | 600 s |
| `runRetentionCleanup` | 900 s |
| `runPayrollAudit` | 900 s |
| `runBookingAudit` | 900 s |
| `runSmsAutoProvisionRetry` | 600 s |
| `runAIDiagnosticScan` (separate interval) | 1800 s |

`resetSecurityCounters` is intentionally **not** locked — it is synchronous, in-memory only, and a double-run is harmless. Locking it would cost a Firestore round-trip for no benefit.

**What does NOT change.** Internal per-job throttles (`_lastReminderRun < 3 * 60 * 1000`, `_lastMembershipRun < 5 * 60 * 1000`, etc.) stay in place as the inner guard. The distributed lock is an **additional outer** guard for multi-instance safety, not a replacement. Job function bodies, setInterval cadence, and single-instance behavior are all functionally unchanged. When only one instance is running (99% of the time right now), the lock always acquires on first try and the jobs run exactly as before.

**What is new.**

- Const `JOB_INSTANCE_ID` at `backend/index.js:33` — unique 16-char hex per Cloud Run revision/instance
- `withJobLock()` helper at `backend/index.js:10283` — non-throwing transactional acquire + release
- Module load prints `[JOBS] Instance id: <hex> · distributed lock active` on first lock acquire so Cloud Run logs show the feature is live
- New Firestore collection `job_locks/` — at most 7 docs at any given time, written ~7 × 4 = 28 ops per 3-minute cycle = ~13k ops/day per instance. Well within free tier baseline

**Rollback.** If anything breaks, revert the commit. Existing jobs go back to unlocked `setInterval` (current behavior). No data migration, no state to clean up except leftover `job_locks/` docs which expire on their own within 30 minutes of the revert.

### Hotfix: Waitlist disappeared from public booking page — SET-006 regression from 849e998 (commit `a3c885f`)

**Symptom:** Owner reported that the `Join waitlist` CTA on the public booking page (`/book/{slug}`) stopped rendering. It was there earlier in the week and then silently vanished after 2026-04-15 backend pushes.

**Root cause** — my own earlier SET-006 fix in commit `849e998`:
- In `/public/config/:workspace_id` I started returning `waitlist_enabled: !!data.waitlist_enabled`. The `!!` coerced *undefined* → *false*.
- In 99% of live Salon+ workspaces the `waitlist_enabled` field was **never explicitly set** in Firestore — the `Settings → Waitlist enabled` toggle defaulted to unchecked in the UI but no one had touched it. So every Salon+ workspace suddenly got `waitlist_enabled: false` from the public config endpoint.
- Frontend at `app/book/[id]/page.tsx:442` uses `setWaitlistEnabled(cfg?.waitlist_enabled ?? !!resolved.waitlist_enabled)`. The nullish coalescing `??` only falls through on `null` / `undefined`, not on `false`. So the explicit `false` from config overrode the plan-feature `true` from `/public/resolve`, and the form disappeared everywhere.

**Two other silent bugs uncovered** during the investigation, both already dead before my 849e998 fix — my change just made one of them visible:
- `POST /public/waitlist/:workspace_id` at line ~9400 rejected submissions with `403 Waitlist is currently disabled` whenever `settingsData.waitlist_enabled` was falsy, including undefined. So even before 849e998, every "click Submit" on an untouched-toggle workspace failed with 403. Form was visible but non-functional. Predates my work.
- `tryWaitlistAutoFill()` at line ~810 bailed immediately on every cancellation if `!settings.waitlist_enabled`. The auto-notify feature (cancel slot → tell waitlist) was effectively dead for every Salon+ workspace that never touched the toggle. Also predates my work.

**Fix applied in `a3c885f`** — one consistent semantic across all three sites: *waitlist is ON by default on any plan that includes the feature; the Settings toggle is an explicit OFF override, not an opt-in.*

1. **`/public/config/:workspace_id`** — only emits `waitlist_enabled` when it is an explicit boolean in Firestore. When absent, the field is omitted from the response, and the frontend `??` guard correctly falls through to the plan-feature value from `/public/resolve`.
2. **`POST /public/waitlist/:workspace_id`** — rejects only when the plan does not include waitlist OR when the owner explicitly set `waitlist_enabled === false`. Undefined (default) passes.
3. **`tryWaitlistAutoFill()`** — bails only on explicit `false`. Plan-level gate is already enforced in the caller path.

**Behavior matrix after the fix:**

| Scenario | Before `a3c885f` | After `a3c885f` |
|---|---|---|
| Salon+ workspace, toggle untouched (default) | Form hidden (regression from 849e998), submit would 403 anyway | ✅ Form visible, submit accepted |
| Salon+ workspace, explicit `waitlist_enabled: true` | Form visible, submit accepted | ✅ Same — unchanged |
| Salon+ workspace, explicit `waitlist_enabled: false` | Form hidden, submit rejected | ✅ Same — explicit off respected |
| Individual plan | Plan gate blocks regardless | ✅ Same — plan gate unchanged |
| Element Barbershop | Not affected by this path | ✅ Not affected |

**Lesson learned.** Boolean defaults in public API responses are load-bearing. ### Smart Booking Audit System (`runBookingAudit()`)
Background job running every 4 hours, scans all workspaces for booking anomalies.

**8 Health Checks:**

| # | Check | Severity | Action |
|---|-------|----------|--------|
| 1 | Double bookings (overlapping times, same barber) | critical | Alert |
| 2 | Ghost barber (deleted/inactive barber on future booking) | critical | SMS + email to client |
| 3 | Stale "booked" status (past >2h) | warning | **Auto-fix to noshow** |
| 4 | Missing client data (no name, phone, or email) | warning | Alert |
| 5 | Orphaned bookings (expired workspace plan) | critical | Alert |
| 6 | Schedule violations (booking outside working hours) | warning | Alert |
| 7 | Cancellation spike (>50% in 24h, min 5 bookings) | warning | Alert |
| 8 | Repeat no-show clients (3+ in 30 days) | info | Alert |

**Notifications:**
- `support@vurium.com` — digest of all critical issues across all workspaces
- Workspace owner — push notification + branded dark-cosmos email
- In-app badge — stored in `settings/booking_audit` doc per workspace

### Auto-Fix Stale Bookings
- Past bookings with `status: 'booked'` (>2 hours overdue) automatically changed to `noshow`
- Adds `auto_noshow: true` + `auto_noshow_at` timestamp for audit trail
- Previously only warned; now self-heals

### Ghost Barber Client Alerts
When audit detects a future booking assigned to a deleted/inactive barber:
- **SMS** to client: "Your specialist is no longer available... rebook here"
- **Email** to client: branded template with "Reschedule" + "Book Another Specialist" buttons
- Dedup via `ghost_barber_notified` flag on booking doc — won't spam on next audit cycle

### Smart Client Recommendations (`getSmartRecommendation()`)
When a public booking fails, the API now returns actionable alternatives:
- On `409 Conflict` — next 3 available slots for same barber + up to 3 alternative barbers
- On `400 Outside Schedule` — next available slots within working hours
- On `404 Barber Not Found` — other active barbers with their nearest open slot

Response includes `recommendation` object:
```json
{
  "error": "Slot already booked",
  "recommendation": {
    "message": "This time is unavailable. Here are the nearest open slots:",
    "alternative_slots": ["2026-04-15T15:00:00Z", "..."],
    "alternative_barbers": [{ "id": "abc", "name": "Mike", "next_slot": "..." }]
  }
}
```

Enhanced endpoints:
- `POST /public/bookings/:workspace_id` (conflict, outside schedule, barber not found)
- `POST /public/manage-booking/reschedule` (conflict, outside schedule)
